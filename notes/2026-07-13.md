## What surprised us

- **LeRobot's Hub cache is not `~/.cache/huggingface/hub`** — it's a separate
  namespace, `~/.cache/huggingface/lerobot/hub`. Anything that assumes the
  standard `huggingface_hub` cache location (e.g. disk-usage checks, cache
  cleanup scripts) needs to look in the right place.
- The metadata-only fetch (`LeRobotDatasetMetadata`) pulled exactly 4 small
  JSON/stats files (136K total) — no video or parquet episode data touched
  disk. Confirms the Hub API genuinely supports a metadata-only path, not just
  "download everything and trust nothing got used."
- `lerobot/pusht` on the Hub: 206 episodes, 25,650 frames, 10 fps. Features are
  `observation.image`, `observation.state`, `action`, plus bookkeeping columns
  (`episode_index`, `next.reward`, `next.success`, etc).
- Installed policy set is broader than the three named in the curriculum
  (ACT, Diffusion, SmolVLA) — `lerobot.policies` core also ships PI0, PI0Fast,
  PI05, Groot, TDMPC, VQBeT, WallX, XVLA, EO1, Evo1, and others, independent of
  which extras were installed. Extras (`pusht`, `diffusion`, `smolvla`) gate
  *simulation environments and heavier deps*, not which policy classes import.
- Only `gym_pusht/PushT-v0` is registered as a gym env — expected, since only
  the `pusht` extra was installed (no `aloha`/`libero`/`metaworld`).
- On macOS, `cv2` and `av` both bundle their own copy of `libavdevice`,
  producing an Objective-C duplicate-class runtime warning
  (`Class AVFFrameReceiver is implemented in both ...`) on every import that
  touches both. Harmless so far, but noisy — worth watching if it ever turns
  into an actual crash rather than a warning.
- torch resolved to `2.11.0` CPU build via the `sys_platform == 'darwin'`
  marker, confirmed via `torch.cuda.is_available() == False` — the
  platform-conditional lockfile approach works as designed on the first try.
