## What surprised us

- **LeRobot's Hub cache is not `~/.cache/huggingface/hub`** — it's a separate
  namespace, `~/.cache/huggingface/lerobot/hub`. Anything that assumes the
  standard `huggingface_hub` cache location (e.g. disk-usage checks, cache
  cleanup scripts) needs to look in the right place.
- The metadata-only fetch (`LeRobotDatasetMetadata`) pulled exactly 4 small
  JSON/stats files (136K total) — no video or parquet episode data touched
  disk. Confirms the Hub API genuinely supports a metadata-only path, not just
  "download everything and trust nothing got used."
- `lerobot/pusht` on the Hub: 206 episodes, 25,650 frames, 10 fps. Features are
  `observation.image`, `observation.state`, `action`, plus bookkeeping columns
  (`episode_index`, `next.reward`, `next.success`, etc).
- Installed policy set is broader than the three named in the curriculum
  (ACT, Diffusion, SmolVLA) — `lerobot.policies` core also ships PI0, PI0Fast,
  PI05, Groot, TDMPC, VQBeT, WallX, XVLA, EO1, Evo1, and others, independent of
  which extras were installed. Extras (`pusht`, `diffusion`, `smolvla`) gate
  *simulation environments and heavier deps*, not which policy classes import.
- Only `gym_pusht/PushT-v0` is registered as a gym env — expected, since only
  the `pusht` extra was installed (no `aloha`/`libero`/`metaworld`).
- On macOS, `cv2` and `av` both bundle their own copy of `libavdevice`,
  producing an Objective-C duplicate-class runtime warning
  (`Class AVFFrameReceiver is implemented in both ...`) on every import that
  touches both. Harmless so far, but noisy — worth watching if it ever turns
  into an actual crash rather than a warning.
- torch resolved to `2.11.0` CPU build via the `sys_platform == 'darwin'`
  marker, confirmed via `torch.cuda.is_available() == False` — the
  platform-conditional lockfile approach works as designed on the first try.

# Module 0 → 2, Session 2: baseline rollout

## What was tried

- Checked the Hub for pretrained PushT checkpoints under `lerobot/`:
  `diffusion_pusht` (~1.05GB), `diffusion_pusht_keypoints` (~995MB),
  `vqbet_pusht` (~158MB). No ACT checkpoint exists for PushT on the Hub — ACT
  checkpoints are trained on ALOHA, which we didn't install the extra for.
  Chose `diffusion_pusht` since it's the policy the curriculum names directly.
- Ran `lerobot-eval --policy.path=lerobot/diffusion_pusht --env.type=pusht
  --eval.n_episodes=5` — hit a hard failure first (see below), fixed via the
  library's own migration script, then re-ran successfully.
- Result: **80% success (4/5)**, avg max reward 0.8, avg sum reward 82.3,
  seed 1000, ~16 min wall time for 5 episodes on CPU/MPS. Episode 1 was the
  one failure (reward 0.0). Videos + `eval_info.json` saved to
  `experiments/000-baseline-rollout/`.
- Wrote up the full mechanism (what PushT is, why Diffusion Policy is slow at
  inference but cheap to train, per-step input/output dimensions) as a
  standalone lesson: [2026-07-13-lesson-diffusion-policy-pusht.md](2026-07-13-lesson-diffusion-policy-pusht.md).

## What surprised us

- **All three official `lerobot/*_pusht` Hub checkpoints predate the current
  processor-pipeline checkpoint format.** `lerobot-eval` against
  `lerobot/diffusion_pusht` failed with a raw 404 trying to fetch
  `policy_preprocessor.json` — a file that checkpoint never shipped, because
  it was uploaded before LeRobot introduced the `PolicyProcessorPipeline`
  format for normalization/pre/post-processing. LeRobot actually has a
  `ProcessorMigrationError` designed to catch exactly this and print the fix
  command, but the detection (`_should_suggest_migration` in
  `lerobot/processor/pipeline.py:1261`) only runs for **local directories** —
  for a bare Hub repo ID it just lets the bare `FileNotFoundError` propagate,
  so the error message gives no hint about what's actually wrong or how to
  fix it.
  - Fix: downloaded the checkpoint locally (`snapshot_download`), ran
    `python -m lerobot.processor.migrate_policy_normalization
    --pretrained-path <local_dir>`, pointed `--policy.path` at the migrated
    copy. Worked cleanly — this is the library's own sanctioned path, just
    not triggered automatically for Hub IDs.
  - This is content for write-up #1: the entire public `lerobot` org's PushT
    checkpoint zoo is currently broken against the pinned `lerobot==0.6.0`
    eval CLI out of the box.
- Real inference time was ~16 minutes for 5 episodes — nowhere near the
  "well under a minute" estimate from the Session 2 candidate proposal. The
  mechanism (denoising requires dozens of forward passes per action chunk,
  not one) is in the lesson note linked above.
